1. **Configuration ownership is still implicit rather than explicit**
   - Epic 8 introduced behavior that now needs a clear model for where configuration lives and which scope wins.

2. **Review caught too many edge cases late**
   - Several important issues were found after implementation rather than being designed into test cases up front.

3. **BDD and Screenplay infrastructure created repeated friction**

   - Epic 8 story completion was clear, but epic-level tracking lagged until retrospective closure.

## Key Insights and Lessons Learned

1. **The project now needs one canonical configuration model**
   - Context directories, CLI overrides, future workspace config, schema defaults, and built-in defaults must be described as a single system.

2. **Compatibility rules must be treated as product behavior, not cleanup**
   - Saved-context envelopes, legacy JSON support, tag normalization, and deterministic selection affect user trust directly.

3. **Shared test infrastructure deserves product-level discipline**
   - BDD runner wiring and Screenplay ability state are part of delivery infrastructure and need explicit standards.

4. **Cross-epic capability growth now requires stronger planning continuity**
   - Once a feature family spans multiple stories, the team needs an explicit model document, not only story-local implementation notes.

## Next Epic Preview (Epic 9: Cascading Configuration System)

Epic 9 introduces global defaults, workspace configuration, schema-level defaults, and priority-based merging. It is directly dependent on Epic 8 because Epic 8 already introduced configuration-sensitive behavior such as context save locations and reusable context conventions.

### Dependencies on Epic 8 Outcomes

- Epic 9 must absorb Epic 8's context directory behavior into a broader configuration model rather than leaving it as a standalone special case.
- Context-related defaults need to fit cleanly into the same precedence ladder as output format, generation defaults, and schema-level overrides.
- Effective-configuration visibility will matter more now because the system has enough moving parts to confuse users without explicit source attribution.

### Preparation Gaps

- Define one canonical precedence model across built-in defaults, global config, workspace config, schema defaults, and CLI flags.
- Clarify which settings belong in CLI-only configuration versus core schema semantics.
- Publish a configuration-scope matrix covering generation defaults, output defaults, context paths, and future extensibility points.
- Decide how `td config show` should explain both value and source of each setting.

### Technical Prerequisites

- Preserve deterministic behavior regardless of which configuration layer supplies a value.
- Keep override behavior explicit and shallow, as already planned for Epic 9.
- Ensure saved-context and load-context behaviors remain backward compatible once configuration becomes layered.
