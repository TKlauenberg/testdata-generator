# Story 12.4: Platform Migration Preparation

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a product manager,
I want data structured for future platform migration,
so that local CLI usage can evolve to centralized platform.

Stories 12.1 through 12.3 already established the local audit foundation: canonical generation metadata, append-only generation history, and immutable pattern-version snapshots keyed by `patternHash`. Story 12.4 should turn that foundation into a deterministic export contract for future platform ingest without pretending that a platform already exists. Keep the scope disciplined: no remote sync, no server APIs, no lifecycle backend, and no second audit system.

The safest implementation path is to export an existing metadata-bearing local artifact rather than rerun generation inside the export command. The current CLI already embeds metadata into generated JSON, CSV, and SQL outputs, and saved contexts already persist metadata in JSON envelopes. Story 12.4 should package those local artifacts together with matching audit information from `.td-history.jsonl` and `.td-pattern-versions` into one platform-ready JSON bundle. If required audit artifacts are missing because generation ran with `--no-history` or predates Story 12.3, the command must fail clearly instead of fabricating lineage.

## Acceptance Criteria

1. Given future platform evolution is planned, when data is generated locally, then metadata includes fields reserved for platform use.
2. Generated data can be exported with `td export --platform-ready`.
3. Platform-ready export includes all metadata required for later platform import.
4. Export format is documented for the platform team.
5. Data includes generation lineage describing what generated what.
6. Context references are preserved in metadata.
7. A local-to-platform migration guide is documented.
8. Unit tests verify export format structure.
9. Gherkin tests verify platform-ready export completeness.

## Dev Notes

### Story Foundation

- Epic 12 is the auditability and platform-readiness epic. Story 12.1 introduced canonical generation metadata, Story 12.2 introduced append-only history, and Story 12.3 added immutable pattern snapshots and `td diff`.
- The remaining gap is packaging those existing local artifacts into a future-proof export contract that a platform ingest pipeline could trust later.
- PRD requirements FR23, FR30, and FR31 require metadata for traceability, future platform lift, and exportable generation history. Story 12.4 is the bridge from local audit data to a portable migration bundle.
- No dedicated UX planning artifact matching `*ux*.md` exists in planning artifacts. Keep the command UX terse, terminal-friendly, and aligned with existing command help and error-output patterns.
- Keep scope honest: the story prepares for platform migration, it does not build platform lifecycle management, remote persistence, or account-level multi-user workflows.

### Technical Requirements

- Treat `GenerationMetadata` in `packages/core/src/common/generationMetadata.ts` as the single canonical metadata contract. Extend it rather than introducing a second metadata shape in CLI code.
- Add platform-reserved fields as an optional nested object so current metadata consumers keep working and future platform-specific additions stay namespaced.
- Preserve the existing `patternHash` and `lineage` semantics from Stories 12.1 and 12.3 exactly. Do not add a second version id, lineage model, or audit hash.
- Preserve context references as explicit metadata, not just implicit behavior. Reuse `parseContextReferenceExpression()` and the analyzer or generator paths that already understand `@context` syntax instead of regex-scanning final output text.
- Define the platform-ready export as a deterministic JSON bundle that includes:
  - the original local artifact payload,
  - the canonical generation metadata,
  - the matching generation-history entry,
  - the stored pattern-version snapshot keyed by `patternHash`, and
  - enough context-reference metadata for a future platform import pipeline to understand external-data dependencies.
- Use embedded artifact metadata plus audit stores to enrich the export. Do not re-run generation during export, and do not rely on current filesystem state of `.td` sources to reconstruct lineage.
- If generation history or pattern snapshots are missing because a run used `--no-history` or occurred before snapshot persistence existed, return a clear error for `--platform-ready` rather than silently omitting required lineage.
- Keep the export contract deterministic for testing: stable key ordering where custom serialization is used, stable entry ordering for lineage and context-reference metadata, and no nondeterministic timestamps beyond the explicit export timestamp field.
- Limit supported artifact inputs to metadata-bearing local outputs the repository actually produces today: generated JSON, CSV, SQL, and saved-context JSON. Avoid broad generic file-import logic unless a real acceptance test requires it.

### Architecture Compliance

- Keep reusable export-envelope construction, metadata decoding, and audit-data assembly in `@testdata-ai/core`; keep command parsing, file-path resolution, stdout/file writing, and CLI help in `@testdata-ai/cli`.
- Do not push platform export logic into output adapters. Adapters remain responsible for generated-data formatting, while export bundles are higher-level audit and migration artifacts.
- Reuse `packages/cli/src/historySupport.ts` for `history.logDirectory` and `.td-pattern-versions` resolution so export, history, diff, and generate stay aligned.
- Follow the same command-registration pattern already used in `packages/cli/bin/td.ts` with `program.addCommand(...)`.
- Avoid opportunistically changing the entire CLI parse lifecycle from `program.parse()` to `program.parseAsync()` unless the new export command actually exposes a repo-wide issue and you validate the full command surface. That concern is broader than Story 12.4.
- Preserve strict TypeScript, ESM-only module boundaries, and export discipline through `index.ts` files.

### Library / Framework Requirements

- Bun remains the runtime and test runner. Use `bun test` and the existing package-scoped scripts.
- Current repo dependencies remain TypeScript `^5.9.3`, Commander `^14.0.2`, and Bun-first filesystem patterns. Upstream Commander documentation is currently on 14.0.3, but this story does not require a dependency bump.
- Commander best practice still favors a dedicated `Command` instance per subcommand and `addCommand()` registration. Reuse that structure for `td export`.
- Bun documentation still recommends `Bun.file()` and `Bun.write()` for optimized file reads and writes, while `node:fs` remains the recommended path for directory operations such as `mkdir` and `readdir`.

- Unit tests must verify export behavior against each supported local artifact shape that currently exists in the repo: generated JSON, generated CSV, generated SQL, and saved-context JSON.
