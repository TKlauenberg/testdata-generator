## Team Discussion Highlights

### What Went Well

**Test Coverage & Fixtures:**

- **Comprehensive test coverage created confidence** (Tobi's highlight)
  - 57+ tests across 5 stories with both unit and BDD coverage
  - Test fixtures reusable across stories (4.2 created, 4.3 and 4.5 reused)
  - Every command has functional validation
  - Fixtures give concrete examples of tool capabilities

- **Fixture reuse created compounding value**
  - Story 4.2 created valid-simple.td, invalid-syntax.td, invalid-semantic.td
  - Stories 4.3 and 4.5 reused them without duplicating test data
  - Consistency across stories ensured by shared test assets
  - Team can "get a feeling for the tool" through real examples

**Framework & Architecture:**

- **Commander.js choice was excellent**
  - Clean API, intuitive options handling
  - Adding new commands trivial (4.2, 4.3, 4.4 all followed same pattern)
  - No regrets on framework selection

- **Build system solved monorepo challenges**
  - Switch from `tsc` to `bun build` in Story 4.1 resolved TypeScript rootDir conflicts
  - Single 86KB bundled executable
  - Clean separation between packages

- **CLI feels professional and usable**
  - All three commands work intuitively
  - Error messages from Story 4.5 genuinely helpful
  - Next steps guidance in Story 4.4 aids user learning

**Process & Quality:**

- **100% Epic 3 action item completion**
  - Foundation patterns documentation paid dividends
  - Test tagging enabled fast feedback loops
  - Prep sprint prevented blockers (Story 4.2 format issue caught early)

- **Code review process caught issues effectively**
  - Build artifacts cleaned up before merging
  - Test quality improvements (exact exit code checks vs. generic "not 0")
  - Documentation gaps identified and fixed

### Challenges & Growth Areas

**UX Specification Gaps:**

- **Epic 4 was first user-facing work** (Tobi's observation)
  - Previous epics (1-3) built core libraries where UX didn't matter
  - CLI commands required visual style, colors, message format decisions
  - Story ACs lacked UX detail beyond functional requirements

- **UX decisions made during implementation, not planning**
  - Story 4.5: Color choices (red/yellow/blue/cyan) made on the fly
  - Story 4.4: Prompt format '(y/n)' vs '(yes/no)' decided during coding
  - Decisions worked out fine, but team "got lucky rather than being deliberate"

- **Team consensus:** User-facing stories need UX specifications beyond functional ACs

**Build System Configuration:**

- **Build artifact cleanup needed in multiple code reviews**
  - Story 4.1: ~100 .js files accidentally committed
  - Story 4.2: Similar cleanup required
  - Root cause: .gitignore insufficient, TypeScript outputting to wrong directories
  - Pattern repeated across stories - same issue multiple times

- **Interactive prompts (readline) trickier than expected**
  - Story 4.4: async/await with readline required careful handling
  - Testing interactive prompts was awkward
  - Pattern not well-documented for future stories

**Testing Quality Standards:**

- **Gherkin feature files incomplete** (Tobi's critical observation)
  - Multiple stories created .feature files but deferred step definitions
  - Story 4.1 explicitly noted "step definitions deferred to future story"
  - Feature files without implementations are documentation, not automated tests
  - AC said "Gherkin tests verify..." but tests weren't executable

- **Definition of done issue**
  - Team treated Gherkin as "A) feature file exists" not "B) feature file + step definitions"
  - Unit tests covered functionality, so Gherkin felt redundant
  - But Gherkin validates user-facing workflows end-to-end - different purpose than unit tests

### Key Insights

1. **Test fixtures create compounding value** - Reusable test assets (valid-simple.td, invalid-syntax.td) saved time across multiple stories and gave team concrete examples of tool capabilities

2. **User-facing work requires UX specifications** - First CLI epic revealed need for visual mockups, color schemes, message formats beyond functional requirements

3. **Code review process catches quality issues** - Build artifacts, test precision, documentation gaps all caught in reviews before merging

4. **Gherkin tests need step definitions to be valuable** - Feature files without executable step definitions are documentation, not automated tests; mandatory enforcement needed

5. **Preparation between epics prevents blockers** - Epic 4 prep sprint caught Story 4.2 format issue and Story 4.4 template scope issue before implementation began

6. **Build system verification prevents repeated issues** - Same .gitignore and tsconfig problems hit multiple stories; checklist needed for new packages

**1. Create UX specification checklist for user-facing stories**

**2. Establish build system setup checklist**
- **Owner:** Charlie (Senior Dev)
- **Timeline:** Within 1 week
- **Priority:** MEDIUM
- **Success Criteria:**

  * Test build verification step (run build, check for stray files)
  * Document in project wiki or docs/
  * Apply to future package creation to prevent repeated build artifact issues

**3. Enforce Gherkin step definitions before code review**
- **Owner:** Bob (Scrum Master - process monitor)
- **Timeline:** Immediate (applies to Epic 5 onward)
- **Priority:** CRITICAL
- **Success Criteria:**
  * Update story completion checklist: unit tests + Gherkin feature file + step definitions implemented
  * Stories cannot move to 'review' status without executable Gherkin tests
  * Add reminder in story creation process
  * Monitor during Epic 5 to ensure compliance
  * If step definitions feel too heavy for certain stories, question whether Gherkin is right approach

### Documentation

**4. Clarify Story 5.5 DSL syntax for weighted selection**
- **Owner:** Alice (Product Owner) + Charlie (tech review)
- **Timeline:** During Story 5.5 creation
- **Priority:** MEDIUM
- **Success Criteria:**
  * Example DSL syntax showing weighted pick usage
  * Clear documentation of how users specify weights
  * Validate syntax fits existing DSL grammar
  * Include in story AC and dev notes
  * Prevent "figure out syntax during implementation" pattern

### Team Agreements

The team committed to the following working agreements for Epic 5 and beyond:

1. **Gherkin tests require step definitions, not just scenario files** - Executable tests mandatory before code review, not deferred to future stories

2. **User-facing stories need UX specifications** - Beyond functional requirements, define visual style, colors, message formats, interaction flows upfront

3. **Build system verification before story completion** - Check .gitignore patterns, verify no stray build artifacts, validate tsconfig settings

4. **Continue prep sprints between epics for dependency review** - Proven to prevent blockers like Story 4.2 format issue and Story 4.4 template scope clarification

5. **Just-in-time development for test infrastructure** - Build step definitions when needed in stories, not speculatively; avoid pre-building infrastructure that might not get used
